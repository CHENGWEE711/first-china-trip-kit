## 1. Objective

This standard makes one separately scheduled publication, its Landing sessions
and downstream funnel actions joinable without storing personal data. It
replaces the current mix of `social`, `video`, `organic_social`,
`organic_video` and `organic` with one controlled vocabulary for all new links.

```text
Campaign attribution granularity: publication level
```

The three attribution identities are deliberately separate:

```text
utm_campaign = stable commercial initiative
utm_content  = creative identity and version
utm_id       = immutable, non-personal publication identifier
```

One creative may be reused, but every separately scheduled publication receives
its own `utm_id`. The private campaign registry maps that public-safe ID to the
corresponding Metricool publication record. Metricool's own post ID remains
private.

The six standard fields are:

```text
utm_source
utm_medium
utm_campaign
utm_content
utm_id
utm_term
```

No custom `fctk_campaign` query field is needed. `utm_id` is a standard GA4
campaign parameter and is the only publication identifier allowed in the public
campaign URL. Provider IDs, Metricool post IDs, operator notes and customer IDs
never appear in the URL.

This revision is the normative Campaign Naming contract. Any earlier Phase 4B
architecture text that describes only five managed campaign fields must be
reconciled with this publication-level contract before implementation begins.

## 3. Character and length rules

| Field | Required | Maximum | Allowed format | Invalid handling |
| --- | --- | --- | --- | --- |
| `utm_id` | Yes for every managed publication | 32 | Registry-issued lowercase ASCII publication ID | Reject |
| `utm_source` | Yes | 32 | Controlled lowercase ASCII token | Reject |
| `utm_medium` | Yes | 32 | Controlled lowercase ASCII token | Reject |
| `utm_campaign` | Yes | 64 | `a-z`, `0-9`, underscore | Reject |
| `utm_content` | Yes for every managed publication | 64 | `a-z`, `0-9`, underscore | Reject |
| `utm_term` | Only for an approved manual paid-search keyword group | 64 | `a-z`, `0-9`, underscore | Reject |

Normalization is deterministic:

1. Unicode-normalize input, trim edges and lowercase it.
2. Transliterate only an explicitly supported operator label; generated public
   values themselves must be ASCII.
3. Convert one or more spaces/hyphens to one underscore.
4. Collapse repeated underscores and remove leading/trailing underscores.
5. Reject an empty result, an over-length result or characters outside
   `[a-z0-9_]`.
6. Reject values resembling an email, URL, phone number, social handle, customer
   identifier, visa answer, route/date or free-form caption.

The builder must return a validation result with field-level errors; it must not
silently map an operator typo to `other`.

`publishId` is an exception to operator-input normalization: it is issued in
canonical form by the registry and must match exactly. The builder validates it
but never lowercases, transliterates or repairs it; a mismatch is rejected so an
immutable publication ID cannot change silently.

### 3.1 Publication ID contract

The private registry issues `publishId`, and the public URL carries the exact
same value as `utm_id`.

Recommended format:

```text
fctk_2026_000127
```

Rules:

- lowercase ASCII using only `a-z`, `0-9` and underscore;
- maximum 32 characters;
- immutable after assignment and never reused, including after cancellation;
- unique across the complete publication registry;
- one ID for one separately scheduled publication;
- generated by the registry, not typed free-form by an operator;
- no email, username, social-account ID, Metricool ID, customer data, traveller
  data or click ID;
- safe to expose in a public campaign URL;
- privately joinable to one Metricool post ID after scheduling or publication.

Google Analytics calls `utm_id` â€śCampaign ID.â€ť This project intentionally uses
that standard dimension as its publication join key while `utm_campaign` remains
the stable commercial initiative. A future cost-import design must map this
project-specific granularity explicitly and must not assume `utm_id` is an ad
platform's campaign ID.

Do not set one static global GA4 `campaign_id`: it can override the Landing
page's `utm_id`. If a later runtime must set `campaign_id` programmatically, the
value must equal the validated publication `utm_id` for that context and must
pass the same stale-field reset tests as the other campaign parameters.

Official references:

- [GA4 campaign URL parameters and `utm_id`](https://support.google.com/analytics/answer/10917952?hl=en)
- [GA4 Campaign ID import mapping](https://support.google.com/analytics/answer/10071305?hl=en)

## 5. Medium taxonomy

| Canonical medium | Use |
| --- | --- |
| `organic` | All unpaid social/community/video posts, regardless of media format |
| `paid_social` | Paid social placement |
| `paid_video` | Paid placement bought as video inventory on an approved video source |
| `cpc` | Manually tagged paid search; chosen to match GA4's Paid Search rules |
| `email` | Newsletter or lifecycle email |
| `referral` | System-classified external referral; not normally authored |
| `direct` | System-classified direct session; never authored |
| `affiliate` | First China Trip Kit outbound partner tracking, not inbound social posts |
| `other` | Reporting fallback after validation/classification, not an operator shortcut |

`social`, `video`, `organic_social` and `organic_video` are legacy aliases. For
new campaigns, normalize all unpaid posts to `organic`; content format belongs in
`utm_content`, not the medium. Historical data is not rewritten in GA4. A
dashboard migration layer may group the four legacy values into a displayed
`organic` channel while preserving a â€ślegacy_mediumâ€ť quality flag.

Do not use `search` as a paid-search medium: GA4's default Paid Search rule
expects a paid/cost-per-click pattern, and Google's manual-tagging examples use
`cpc`. Ordinary organic Google/Bing search relies on the referrer/auto
classification and receives no manually authored UTM.

### 5.1 GA4 classification risk

GA4 Default Channel Group can classify an unrecognized source with medium
`organic` as **Organic Search**, because `organic` is an Organic Search medium
rule while Organic Social and Organic Video also depend on Google's maintained
source-category lists. Therefore `utm_medium=organic` is retained only under the
following mandatory controls:

1. a versioned, source-first internal reporting semantic layer;
2. a matching GA4 Custom Channel Group and, only after review, an optional
   Primary Channel Group;
3. Production classification tests for every approved source;
4. an explicit `other` fallback for unknown or incompatible source/medium pairs;
5. a data-quality comparison and alert whenever the raw GA4 channel differs from
   the expected internal group.

The GA4 Default Channel Group remains visible as a diagnostic field. It is never
blindly copied into the displayed First China Trip Kit reporting group.

### 5.2 Source-first internal reporting groups

Rules are evaluated top-to-bottom against the controlled source and medium
values. The displayed reporting group is the result below:

| Internal reporting group | Approved source/medium rule |
| --- | --- |
| `paid_search` | `google` or `bing` + `cpc` |
| `email` | `newsletter` + `email` |
| `paid_video` | `tiktok`, `youtube` or `twitch` + `paid_video` |
| `paid_social` | `instagram`, `threads`, `bluesky`, `pinterest`, `reddit`, `quora` or `facebook` + `paid_social` |
| `organic_video` | `tiktok`, `youtube` or `twitch` + `organic` |
| `organic_social` | `instagram`, `threads`, `bluesky`, `pinterest`, `reddit`, `quora` or `facebook` + `organic` |
| `affiliate` | Approved provider-owned outbound or affiliate aggregate; never inferred from a public inbound campaign URL |
| `referral` | System-classified approved external referral + `referral` |
| `direct` | No qualifying campaign/referrer + `direct` |
| `other` | Unknown source or any incompatible source/medium pair |

An approved source that GA4 does not recognize still uses its expected internal
group. For example, an approved social source plus `organic` remains
`organic_social` internally even if GA4 reports Organic Search. That difference
must produce a mismatch record; the implementation must not rewrite GA4's raw
historical value.

Before an approved source can be used in a scheduled campaign, a Production-host
classification test must confirm the Landing session's source, medium,
`utm_id`/Campaign ID and raw GA4 channel. The source-first resolver must return
the expected group. A failing or missing test blocks scheduling for that source.
The data-quality view reports mismatch count, mismatch rate, affected source and
publication ID without exposing a click ID or provider identifier.

Google Ads auto-tagging is a separate acquisition design from manually authored
campaign URLs. Section 13 defines the complete `gclid`, `fbclid` and `ttclid`
boundary.

## 6. Campaign naming

Pattern:

```text
{initiative}_{optional_market_or_theme}_{optional_phase}
```

Rules:

- Use a stable campaign for one commercial objective across platforms.
- Do not put platform, content format or individual post number in campaign;
  those belong in source/content.
- Do not encode dates unless the initiative genuinely expires.
- Do not rename a campaign after links have been used.
- A phase suffix such as `_p2` is allowed only when the measurement objective or
  offer changes materially and the phase start is recorded in the campaign
  registry.
- Evergreen content uses an approved stable name such as
  `china_first_trip_evergreen`, not a new monthly campaign.

Current grandfathered campaign:

```text
china_first_trip_launch
```

Keep this exact value for the current launch links. If a future Phase 2 is
approved, register `china_first_trip_launch_p2` before scheduling; never mix the
new name into existing posts retroactively.

## 7. Content identifier naming

Pattern:

```text
{topic}_{format}_{hook_or_angle}_{version}
```

Recommended controlled topic tokens:

```text
payment
apps
visa_transit
checklist
arrival
internet
transport
```

Recommended format tokens:

```text
text
carousel
pin
short
video
story
bio
answer
comment
email
```

Version is two digits (`01`, `02`). The identifier describes the creative, not
the person or account that published it.

Examples:

```text
payment_text_credit_card_hook_01
payment_carousel_backup_layers_01
apps_pin_prepare_eight_tools_01
visa_transit_short_third_country_rule_01
checklist_email_preflight_01
```

Do not copy the full caption into `utm_content`. Do not use usernames, social
post IDs that expose account identity, email addresses or traveller details.

## 8. Optional term convention

`utm_term` is omitted for normal organic posts. Use it only when an approved
manual paid-search campaign needs a controlled keyword group, for example:

```text
first_china_trip
china_payment_apps
visa_free_transit
```

It must not contain a searcher's raw query, audience email, lookalike-list name,
demographic detail or free text. Paid social audience groups stay in the private
campaign registry; they are not exposed through `utm_term`.

## 9. Original, repost and cross-post conventions

| Scenario | Source / campaign | Creative (`utm_content`) | Publication (`utm_id`) |
| --- | --- | --- | --- |
| Original platform post | Actual platform; stable campaign | New creative ID ending in `_01` | New unique `publishId` |
| Edited repost on same platform | Same platform and campaign | Increment version, e.g. `_02` | New unique `publishId` |
| Exact repost on same platform | Same platform and campaign | Reuse only when creative, CTA and destination are identical | New unique `publishId`; never reuse the earlier publication ID |
| Cross-post of the same creative | Each actual platform; same campaign | Reuse the same creative ID if the asset, CTA and destination are unchanged | One new unique `publishId` per platform publication |
| Platform-specific adaptation | Actual platform; same campaign | Use a distinct format/angle/version token | New unique `publishId` |
| Paid amplification | Actual platform; same campaign unless objective changes | Reuse only for an identical creative; otherwise version it | New unique `publishId`; use `paid_social` or `paid_video` according to the approved source group |
| Evergreen resurface | Actual platform; evergreen campaign | Increment version when creative, hook, CTA or destination changes | New unique `publishId` for every resurface |
| Link correction after publication | Actual platform; same campaign unless objective changes | Increment only when creative, CTA or destination meaning changes | Mark the prior record `superseded` and issue a new `publishId`; never edit the published query string |

Changing only a pre-publication schedule may retain a reserved `publishId` when
the source, medium, Landing, campaign, creative and URL are unchanged. Once a
publication has gone live, its ID, campaign dimensions, publication kind,
lineage and public URL are immutable; lifecycle status may advance through the
controlled state machine. Cancelled IDs and IDs involved in a correction are
never recycled.

Metricool's own post identifier stays private. The registry maps it one-to-one
to the public-safe `publishId`; it is never copied into `utm_id`, another UTM,
GA4, a public report or operator-facing campaign URL.

## 12. Future campaign URL contract

Proposed location: `lib/growth/campaign-url.ts`.

```ts
type CampaignUrlInput = {
  landing: "pay_in_china" | "china_visa_free" | "china_checklist";
  publishId: PublishId;
  source: CampaignSource;
  medium: CampaignMedium;
  campaign: string;
  content: string;
  term?: string;
};

type CampaignUrlResult =
  | { ok: true; url: string; normalized: CampaignUrlInput }
  | { ok: false; errors: Array<{ field: string; code: string }> };
```

Required behavior:

- Resolve Landing names from an internal allowlist, never from arbitrary input.
- Require a registry-issued, unique `publishId` and map it exactly once to
  `utm_id`; the builder never derives it from source, timestamp or Metricool ID.
- Generate HTTPS URLs on the canonical domain.
- Parse existing query parameters and remove all existing `utm_*` values before
  writing exactly one normalized value per approved field, including `utm_id`.
- Reject unexpected existing query keys unless the Landing explicitly permits
  them; never preserve an email, route or free-text query.
- Never generate, accept as authored input or fabricate `gclid`, `fbclid` or
  `ttclid`. Platform-decorated inbound URLs are governed separately by Section
  13 and are not passed back through the campaign builder.
- Use `URL`/`URLSearchParams` encoding, with deterministic order:
  `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`, `utm_id`, then
  optional `utm_term`.
- Return normalized data and field-level errors for operator review.
- Never shorten the link using an unapproved third-party service.
- Log no secrets or user data.

Minimum future tests:

1. Each approved Landing and source/medium combination.
2. Unknown Landing, domain, scheme, port and credentials rejection.
3. Case/space/hyphen normalization.
4. Unicode, empty, over-length and invalid-character rejection.
5. Email, URL, phone-like and free-text PII-pattern rejection.
6. Existing/duplicate UTM removal produces exactly one of every required field,
   including one `utm_id`.
7. Duplicate publication IDs are rejected by the registry and builder boundary.
8. An exact same-platform repost reuses `utm_content` but receives a different
   `utm_id`.
9. A cross-platform reuse of one unchanged creative keeps `utm_content`, changes
   source and receives one unique `utm_id` per publication.
10. Correct encoding and deterministic parameter order.
11. `utm_term` is omitted when empty.
12. Internal-link helpers refuse to add UTM or click-ID values.
13. Six examples above snapshot to their exact URLs and distinct publication IDs.
14. Google/Bing + `cpc` is accepted for manual paid search; Google/Bing +
    `search` is rejected.
15. Every GA4-recognized approved source resolves to its expected internal group.
16. A simulated GA4-unrecognized approved social/video source with `organic`
    still resolves source-first to the expected internal group while recording a
    raw-GA mismatch.
17. An unknown source or source/medium mismatch resolves to `other`, raises a
    data-quality alert and cannot be scheduled.
18. `gclid`, `fbclid` and `ttclid` are preserved byte-for-byte by the inbound
    Landing/owned-redirect path until the approved vendor read; none is produced
    by the builder or copied into analytics/event payloads.
19. A Google Ads auto-tagged link remains free of manually authored UTMs unless
    a later reviewed reconciliation contract explicitly allows them.
20. A registry record joins public `utm_id` to exactly one private
    `metricoolPostId` without exposing that provider ID.
21. A publication-level report reconstructs original, exact repost and
    cross-platform rows from GA4 Campaign ID (`utm_id`) plus the registry's
    explicit publication kind and lineage, without a source/content/time
    heuristic.
22. Registry validation rejects missing lineage targets, self-references,
    cycles, invalid publication-kind/lineage combinations and inconsistent
    correction/supersession status.

## 13. Click-ID handling

`gclid`, `fbclid` and `ttclid` are platform-generated click identifiers. They
are not campaign registry IDs, publication IDs or UTM values.

| Click ID | Platform boundary | First China Trip Kit rule |
| --- | --- | --- |
| `gclid` | Added by Google Ads auto-tagging | Builder never generates it; owned redirects and the Landing must preserve an incoming value unchanged until the approved consent-aware Google tag can read it |
| `fbclid` | Reserved inbound Meta click parameter | Builder never generates it; preserve unchanged for an approved consent-aware Meta integration, whose active configuration must be verified separately |
| `ttclid` | Added to TikTok ad Landing URLs | Builder never generates it; preserve unchanged until the approved consent-aware TikTok tag can read it |

The campaign builder and Landing runtime have different responsibilities:

- The builder emits a clean, deterministic UTM URL and rejects caller-supplied
  click IDs as `reserved_click_id_not_authored`.
- An advertising platform may decorate that URL after a real click. The inbound
  Landing route, middleware and owned redirect chain must not strip, lowercase,
  decode/re-encode, rewrite or replace the value before the relevant approved
  platform tag has had an opportunity to read it.
- The initial architecture performs no address-bar cleanup. Any later cleanup
  requires a consent/privacy review and a platform-specific network test.
- Click IDs never enter `utm_id`, the campaign registry, campaign session
  storage, GA4 custom event parameters, Newsletter/Brevo fields, Metricool
  metadata, sitemap/canonical URLs, ordinary application logs or dashboard
  dimensions.
- The application does not copy them onto internal or outbound links. A future
  server-side conversion/offline attribution pipeline requires its own approved
  access, retention and consent contract.
- Preserving an ID does not guarantee attribution; consent, platform settings,
  browser behavior and provider attribution windows still apply.

Google Ads modes are mutually exclusive until a later reconciliation design is
reviewed:

```text
Auto-tagging:
clean approved Landing URL + platform-generated gclid
no manually authored UTM set

Manual campaign URL:
approved UTM set + publication-level utm_id
no operator-generated gclid
```

Do not append manual UTMs to an auto-tagged Google Ads link. Meta/TikTok may
append their click ID to an already approved manual campaign URL, but First China
Trip Kit never creates the click-ID value.

Official references:

- [Google Ads auto-tagging and GCLID](https://support.google.com/google-ads/answer/1752125?hl=en)
- [Meta `fbp`/`fbc` parameters and inbound `fbclid`](https://developers.facebook.com/docs/marketing-api/conversions-api/parameters/fbp-and-fbc/)
- [TikTok Click ID](https://ads.tiktok.com/help/article/tiktok-click-id?lang=en)

Proposed record:

```ts
type CampaignPublication = {

  publishId: string;
  publicationKind:

    | "exact_repost"

    | "platform_adaptation"
    | "paid_amplification"

    | "scheduled"

    | "cancelled"
    | "superseded";
  scheduledAt: string | null;
  publishedAt: string | null;
  publicPostUrl: string | null;

```

Public campaign URL mapping:

| Registry field | Public parameter |
| --- | --- |
| `campaignId` | `utm_campaign` |
| `publishId` | `utm_id` |
| `source` | `utm_source` |
| `medium` | `utm_medium` |
| `content` | `utm_content` |
| `term` | `utm_term` when present |

`campaignId` is the stable commercial initiative ID and must equal the approved
`utm_campaign` value. `publishId` is the immutable publication ID and must equal
`utm_id`. `publicationKind`, `parentPublishId`, `supersedesPublishId`,
`metricoolPostId` and `operatorNotes` are private. `status`, timestamps and
`publicPostUrl` remain in the private registry; a public post URL is not a
campaign parameter and must contain no stored click ID.

Registry rules:

- one separately scheduled publication equals one `publishId`;
- `publishId` uniqueness is enforced across every status and IDs are never
  deleted/reused merely because a post was cancelled or corrected;
- original posts, exact reposts, paid amplifications and cross-platform posts
  each receive a new `publishId`;
- `publicationKind` records the controlled publication relationship explicitly;
  reports must not infer it from source, content or timestamps;
- an original publication has `parentPublishId: null`; a repost, cross-post,
  adaptation, amplification or resurface references the existing source
  publication in `parentPublishId`;
- a correction sets `supersedesPublishId` to the prior published record, the
  prior record moves to `superseded`, and the new record follows the normal
  `draft` â†’ `scheduled` â†’ `published` lifecycle; all other publications use
  `supersedesPublishId: null`;
- lineage references must resolve to an existing record, cannot reference the
  same `publishId`, cannot form a cycle and are immutable after publication;
- allowed status transitions are `draft` â†’ `scheduled` â†’ `published`,
  `draft|scheduled` â†’ `cancelled`, and `published` â†’ `superseded`; a status
  transition is audited and never changes attribution, identity or lineage
  fields;
- an unchanged creative may reuse `content` across publications;
- any creative, hook, CTA or meaningful destination edit creates a new
  `utm_content` version as well as a new publication ID;
- a link correction after publication creates a new publication ID and, where
  the creative/CTA/destination meaning changed, a revised content version;
- no customer, traveller, subscriber or social-account identity is allowed;
- `metricoolPostId` may be populated after scheduling, but must map to no more
  than one `publishId`; a publication cannot map to multiple Metricool posts;
- all timestamps use ISO 8601 with an explicit offset; missing lifecycle values
  are `null`, never guessed.

Publication-level reporting joins GA4's Campaign ID dimension (populated from
`utm_id`) to registry `publishId`, then privately resolves Metricool performance
through `metricoolPostId`. If Campaign ID is missing or duplicated, show the row
as unattributed/invalid. Do not reconstruct a publication by combining source,
content and timestamp heuristically.

## 16. Pre-scheduling validation checklist

- [ ] Destination is one of the three approved Landing routes.
- [ ] HTTPS and canonical `www` host are present.
- [ ] Source matches the actual publishing platform.
- [ ] Unpaid post uses `organic`.
- [ ] Paid social uses `paid_social`; paid video uses `paid_video`.
- [ ] Manually tagged paid search uses `cpc`; ordinary organic search has no
      manually authored campaign URL.
- [ ] `utm_term`, when present, is a registered paid-search keyword group and
      contains no raw query or audience information.
- [ ] Campaign is registered and unchanged.
- [ ] `utm_id` equals one unique, immutable registry `publishId` and has never
      been used for another scheduled publication.
- [ ] Content ID is unique for a changed creative/CTA/destination.
- [ ] Exact repost/cross-post reuses content only when unchanged and still has a
      new publication ID.
- [ ] No personal data, full caption, account handle or traveller answer appears.
- [ ] There is one instance of each UTM and no unrelated query parameter.
- [ ] The builder did not generate `gclid`, `fbclid` or `ttclid`.
- [ ] This source has a passing Production source/channel classification test.
- [ ] The URL returns HTTP 200 and the canonical remains the clean Landing URL.
- [ ] Mobile page, primary CTA and downstream Hub work.
- [ ] The operator records the Metricool schedule/post reference privately.

## 17. Historical migration

1. Freeze the current CSV as a legacy artifact; do not rewrite already-published
   URLs.
2. Register `social`, `video`, `organic_social` and `organic_video` as legacy
   aliases in the reporting normalization layer.
3. Generate all new links with canonical `organic`, `paid_social`,
   `paid_video`, `cpc` or the other approved medium matching the real channel,
   plus a unique `utm_id`.
4. Add a dashboard data-quality row for legacy-medium share and missing content.
5. Compare old and new periods separately until two weeks of clean data exist.
6. Retire `scripts/build-utm-url.mjs` and the old `lib/utm.ts` builder only after
   the future shared utility passes unit/CLI tests and all operator docs point to
   one implementation.
7. Treat historical rows without `utm_id` as creative/campaign-level legacy;
   do not fabricate publication IDs or claim post-level reconstruction.

## 18. Governance ownership

Live campaign registry records belong in access-controlled operational storage
that is excluded from the browser bundle, static exports and any public source
artifact. A future `data/growth/campaigns.ts` may define reviewed schemas,
controlled vocabularies and non-sensitive campaign metadata, but it must not
contain live `metricoolPostId` values, operator notes or other private registry
records. Adding a source, medium, campaign or Landing requires review, matching
tests and an operator-doc update. Publication records must pass uniqueness,
privacy, lifecycle and join validation before a URL can be scheduled. Metricool
remains the publisher; this standard governs links only and does not authorize
post scheduling in the architecture phase.

## 19. Final recommendation

```text
Campaign attribution granularity: publication level
```

`utm_campaign` identifies the stable commercial initiative. `utm_content`
identifies the creative and its version. `utm_id` identifies one immutable
publication. The private registry maps that publication to Metricool scheduling
and public-post records without exposing Metricool/provider identifiers.
